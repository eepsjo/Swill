    代理中的 ECH¶

    文章将讲解 vless + tls(ech) + ws 的详细流程，并解释 vless 协议的某些核心点。

    先说结论：给代理套 ech，只需要代理本身开启 ech 即可；换言之 ech 的正确用法就是隐藏代理 sni 而不是在浏览器中开启。 在 cf worker 搭建的节点甚至无需修改服务端即可使用 ech。

    网上所谓的给自己的域名套 ech 纯粹脱裤子放屁/想搞慈善/想搞蜜罐；而托管在 cf 的域名并不是所谓天生支持 ech，而是由于 cf 共享 ech config，使得客户端可以请求任一支持获取 ech config 的域名来获取这个共享的 config。

    具体来说是从 HTTPS 的 dns 记录中获取，如 https://dns.google/resolve?name=cloudflare-ech.com&type=HTTPS 可以获取到
    {"Status":0,"TC":false,"RD":true,"RA":true,"AD":false,"CD":false,"Question":[{"name":"cloudflare-ech.com.","type":65}],"Answer":[{"name":"cloudflare-ech.com.","type":65,"TTL":300,"data":"1 . alpn=h3,h2 ipv4hint=104.18.10.118,104.18.11.118 ech=AEX+DQBB7AAgACAIhfOikk0PVYN6CwzRQuwctTYT4Sr+Csruv3Dqe/19ZwAEAAEAAQASY2xvdWRmbGFyZS1lY2guY29tAAA= ipv6hint=2606:4700::6812:a76,2606:4700::6812:b76"}],"Comment":"Response from 173.245.58.158."}
    其中“AEX+”到“=”的内容即为 config。注意到结尾为“=”，系 base64 加密，解码后可发现内容包含 cloudflare-ech.com，乃 ech 模拟之 sni。

    以下列出几个可获取 cf ech config 的域名：
    cloudflare-ech.com（最标准的）
    cloudflarestaging.com（东大有备案，推荐）
    cloudflarechallenge.com
    cloudflare.dev
    research.cloudflare.com
    pq.cloudflareresearch.com
    gitlab.io
    www.nslookup.io
    esm.sh
    上文“慈善/蜜罐”说出自此：理论上你可把自己托管在 cf 的域名变成可以获取 ech 的域名，而不使用这些。但我们的目的是藏好自己，用自己的域名本身就违背了通过 ech 隐藏 tls 握手中明文的 sni 的初衷。

    注意：“共享 ech config”意味着不管你用了上面的哪个域名（在同一时间）获取的 config 都是一致的，即伪装的 sni 都是cloudflare-ech.com，上述几个域名的唯一作用就是对它们发起 DoH 来获取 config。

    注意第二弹：shadowrocket 暂时不支持动态获取 config。mihomo 可以直接在相关键值填入上述域名的任一，只要配置的nameserver支持 DoH（且不污染上述域名（所以我推荐备案的那个））即可动态获取 config。
    之所以要动态获取是因为这个 config 也会随时间轮替。

    现在将基于 vless + tls(ech) + ws 模拟一次完整的代理请求。

    首先是 server 参数，也就是我们填写优选域名/IP 的地方，理论上由于 Anycast IP 机制此处可填写任意托管在/使用 cf cdn 的域名，如www.visa.com或我们自己的域名。填写域名时，客户端会先解析这个域名的 dns 的 A/AAAA 记录，以此获得一个 cf ip（v4/6），并通过这个 ip 去尝试向服务端发送请求。

    基于 TCP/IP 的互联网，ip 是唯一门牌号；所谓优选域名只是“能快速解析出适用于当前运营商环境的 cf ip”的域名而已；但优选 ip 是真正的“优”，能提升网络速度（不需要 dns 解析）。
    由于国内有三网，每个网络对特定 cf ip 的连接速度是不同的，因此很难找到多网优化的 ip，这也是为啥 wifi 能用蜂窝数据就卡，极有可能是运营商不同。
    网上很多优选域名是站长自己维护站点 dns 记录的，确保解析出来一定是优选 ip。好的站点可以解决切换运营商就要换 ip 的痛点，缺点是 dns 解析本身要时间/留痕迹。

    最近在使用一个优选域名*.tencentapp.cn为什么是通配符？因为此站点运用了“泛域名”技术，任意前缀请求出来的都是同样的记录。

    一旦确立了连接目标 ip 就来到第二步：发起 tls 握手确定身份。ech 在此处的功能就是把 sni 伪装成cloudflare-ech.com，在 isp/gfw 看来你就是在请求这个域名；但同时在 tls 握手包中保留了一些信息，包括你的真实 sni。

    传统的 tls 握手中，由于 sni 是明文的，cf 能据此知道你的流量去向。ech 加密后，流量来到 cf 边缘节点，cf 发现域名系 ech 域名后会尝试解析 tls 包并最终解开加密获取真实 sni（中间人无法攻击，有非对称加密等，可自查原理），也能知道你的流量去向，帮你转发。服务端看到的和 ech 加密前完全一致，因此无需修改服务端；客户端只需要指定 ech 功能，tls 握手的 sni 不用更不能改。

    我们的访问代理时流量都在代理内部，无需给域名本身套 ech 功能。这就是文章开头结论的由来。

    接下来才是传输层协议，实际上此处无论 ws 还是 gRPC 还是 xhttp 都一样，通过配置好的路径和 uuid 确认身份就算完成代理连接了。

    这里额外讲一下 reality 为啥不能套 cdn。
    由于 reality 使用 xtls 模拟完整 tls 握手，ech 无需配置更无效。
    具体来说 reality 没有传输层协议，而是使用 tcp（或者叫 raw 传输）和 vless 自研的模拟 tls 的 xtls 协议；后面出的 vision 流控系 xtls 防止被 gfw 识别的小把戏。一旦使用其他传输层，xtls 就无法启用（虽然是现在是需要用 tls 的，但是也能不开。不开就是裸奔）。
    所谓的模拟就是连 sni 都是偷别人证书来模拟的。都这样了还需要 ech 干嘛呢。

    关键点复盘：

    1. server是定义连接到服务器的参数，本案例中可用任意 cf 域名或 ip；
    2. sni或servername是 tls 握手时用来确定身份的参数，必须与服务端配置的参数一致；ech 配置独立于 tls，能伪装 tls 明文的 sni 但不会修改真实 sni；
    3. host是传输层协议内的参数，也需要和服务端一致。

    题外话：vless 神话¶

    本来是打算展开聊聊的，但是觉得 RPRX 不值得我单独为他写一篇文章。

    如果你曾经用过 ai，或逛过博客论坛，当你试图搜索 vless 之外的协议时，你最终会看到“为什么不用 vless”之类的结果；当你搜索非 xhttp 的传输层协议时，你会看到无数描述 xhttp 优点的结果……似乎最优解是 Reality。但真的是这样的吗？

    RPRX 的为人和言行这里不过多赘述，我就网络上传得比较欢的几个协议焦虑问题展开：

    1. ss/ssR 全随机数问题；
    2. trojan 的 TiT（TLS in TLS，不是奶子）问题；
    3. vmess 的复杂加密问题；
    4. hy2 等 udp 系协议的 QoS 问题，

    等等等等。

    这里需要引入重要概念：所以代理协议都需要包装到合法的传输层协议，否则特征明显（Reality 的 XTLS 及其流控解决了传输层特征，但其他特征是无法避免的，因为其本质就是直连协议），而 TLS 是确保传输安全的，前文的 vless + ws + tls 中 vless 可以替换成 trojan；理论上 Reality 也可以给 hy2 用。

    第二个重要概念：市面上的内核几乎全部尝试自己重新实现协议，也就是说，协议本身是一种“通用”（在代理核心中）的“语言”，一种概念，而不是实体。

    代理协议应该以两点为核心：一是隐蔽，一是安全；它们的重点应该是客户端安全，而非服务端。总结就是，不能让使用“魔法”的事实泄露给第三方。

    vless 的神话在于过分丑化其他传统协议的同时过分宣传自己的解决方案，但自己的协议本身也有很多无法处理的特征；如果全天下都用一个协议，那就不存在隐蔽性了。而上文案例使用 cf ip 和 ech 完全伪装了服务端，trojan + ws 一定程度上解决 trojan 的 TiT 问题，何不尝试呢？

    不要过分相信 RPRX 和他的宗教：xray 内核社区，但也不要轻视他；同时保持对其他协议的包容态度。人话，做个理中客。
