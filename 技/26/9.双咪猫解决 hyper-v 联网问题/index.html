    双咪猫解决 hyper-v 联网问题¶

    核心思路：宿主机开启 allow-lan，配置 mixed-port 和代理鉴权；虚拟机 mihomo 全局模式，通过 socks 代理路由到宿主机虚拟网卡（hyper-v 的 defalut-interface，不是 tun）。

    好吧……实际上宿主机 mihomo 不用 fake-ip 的话就没这些事……唱歌盒子你赢了。

    现在关闭虚拟机的 mihomo，或把它改成直连，虚拟机的软件发起 DNS 请求，往往会被宿主机的 tun 返回 fake-ip 或直接丢弃数据包（如果宿主机没开启 allow-lan）。

    因为宿主机开启的是 fake-ip，加之 hyper-v 的虚拟网卡在逻辑上是 tun 的“网上邻居”，因此来自虚拟机的流量是“局域网其他设备”的，如果（或者说，通常？）这些流量中包含 IP 请求，宿主机 mihomo 需要开启嗅探器，但是我不想开啊兄弟，所以只要让来到宿主 tun 的请求全部都是域名就行。

    在虚拟机开启第二个 mihomo，形成双咪猫，并用文章最前面写的核心思路，即可确保流量全部进入宿主机 tun，并享受宿主机的分流。

    几个坑和解决方案¶

    1. 一直开启 allow-lan 增加了攻击面 😥
      \(\to\) 通过 webui 实现按需开启。
    2. mixed-port 最好自定义不要用默认的，这也是减少攻击面的做法。
    3. 同 2，一定要开启鉴权！
    4. hyper-v 的虚拟网卡的网关每次宿主机重启都会变，我在网上查到可以用命令改变它的行为让它固定，但是我不想这么做 😭
      \(\to\) 在虚拟机 mihomo 启动前执行脚本检测当前默认网关（对应宿主机的虚拟网卡的 IP）并写入节点（文章后面有示例喵～(∠・ω< )⌒☆

    示例¶

    宿主机配置（部分）： 

    authentication: ["【用户】:【密码】"]
mixed-port: 【端口】

    虚拟机配置（部分）：

    这里把节点直接写到 config 里面了；同时这个配置有分流，先处理了一遍需要拒绝的，免得流量进入宿主机。不过感觉没啥必要。

    tun:
  enable: true
  device: utun
  stack: mixed
  dns-hijack: [any:53, tcp://any:53]
  auto-route: true
  strict-route: true
  auto-detect-interface: true
  disable-icmp-forwarding: true

dns:
  enable: true
  ipv6: false
  prefer-h3: false
  enhanced-mode: fake-ip
  fake-ip-range: 198.18.0.1/16

proxies:
  - name: 【代理的名字】
    type: socks5
    server: 【hyper-v 网关】# 下面写了怎么动态获取
    port: 【端口】# 这里是上面宿主机配置里的。下同
    username: 【用户】
    password: 【密码】

proxy-groups:
  - {name: PROXY, type: select, proxies: [【代理的名字】]}
  - {name: GLOBAL, type: select, proxies: [PROXY], hidden: true}

rule-providers:
  r: {type: http, behavior: classical, format: text, url: https://raw.githubusercontent.com/eepsjo/0/refs/heads/0/r}
  R: {type: http, behavior: domain, format: yaml, url: https://anti-ad.net/clash.yaml}

rules:
  - AND,((NETWORK,UDP),(DST-PORT,443)),REJECT
  - RULE-SET,r,REJECT
  - RULE-SET,R,REJECT
  - MATCH,PROXY

    用于动态获取网关 IP 并写入同目录下 config 的 PowerShell 脚本：

    必须在 mihomo 运行前/未运行时才能获取真实正确的 IP。

    $gateway = (Get-NetRoute -DestinationPrefix 0.0.0.0/0).NextHop | Select-Object -First 1
(Get-Content config.yaml) -replace 'server: .*', "server: $gateway" | Set-Content config.yaml

    启动脚本：

    只是多了一行启动 ps1 然后判断真假？！ 的逻辑。vbs 必须用任务计划程序才能以管理员启动喔。

    Set shell = CreateObject("WScript.Shell")
shell.Run "powershell.exe -ExecutionPolicy Bypass -File .\server.ps1", 0, True
shell.Run ".\mihomo【的名字】.exe -d .\", 0
