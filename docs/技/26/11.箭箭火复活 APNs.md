## 编辑参数

### apnS

可填入一个或多个代理节点或分组的名称，每个节点或分组之间用英文逗号分隔。不填则使用缺省值 `PROXY,DIRECT`，建议填写一个或多个稳定、低延迟的节点，然后在末尾填写 `DIRECT`。

> 手动指定固定的节点，目的是防止切换配置时/代理组切换节点时打断 APNs 长连接导致信息接收延迟。<br>填写直连作为回落兜底，是防止节点全部超时导致 APNs 完全失效（至少直连时不需要绕过 APNs 屏蔽的应用还能收到消息）。

### init-?01st.push.apple.com

是否代理 `DOMAIN-WILDCARD,init-?01st.push.apple.com`。默认关闭（开启表示代理，关闭以切换到直连。下同）。

在兜底代理配置的 [D 规则集](https://raw.githubusercontent.com/Loyalsoldier/clash-rules/refs/heads/release/direct.txt)中存在两条直连规则

``` yaml
payload:
  - 'init-p01st.push.apple.com'
  - 'init-s01st.push.apple.com'
```

与下一条参数冲突，故创建此选项让用户自行定夺。

### +.push.apple.com

是否代理 `DOMAIN-SUFFIX,push.apple.com`。默认开启。

### IPall & IPs

是否代理 APNs 的 IP 段。默认开启 `IPall`。

参考 [Apple 的官方文档](https://support.apple.com/en-us/102266)，这里说明了 APNs 的 IP 段。

`IPall` 为文中属于 Apple 的一个 IP 段 `17.0.0.0/8`，`IPs` 则是文中给出的更小的规则集。`IPall` 可以降低匹配规则的时间，放在了上面，因此若要使用 `IPs` 需关闭 `IPall` 并打开 `IPs`。

> 使用 `IPall` 时网络行为更接近兜底代理。

---

模块配置完成后，开启关闭飞行模式一次（这是网传的方法，目的是彻底关闭再开启一次箭箭火隧道）即可复活 APNs。

## 原理

基于笔者对 APNs 屏蔽的技术原理的*猜测*：*笔者认为*，APNs 通过检测用户的 IP，再根据应用的 ID 来定向屏蔽消息推送。
