# 箭箭火复活 APNs

> 如果本文所述的功能在你的箭箭火里没有，可能是版本太低了。

给出一[模块](https://raw.githubusercontent.com/eepsjo/s/refs/heads/m/apns)用于代理 APNs 流量；也可以试试[精简版](https://raw.githubusercontent.com/eepsjo/s/refs/heads/m/apns-ez)。

> 精简版仅代理 `*.push.apple.com` 域名，而原版模块则在此基础上参考了 [Apple 的官方文档](https://support.apple.com/en-us/102266#:~:text=assigned%20to%20Apple.-,If%20you,-can%27t%20allow%20access)，将可能用于 APNs 的 IP 段全部代理，来应对可能的硬编码 IP 或 DNS 污染情况。

安装模块后可以根据自己的网络环境决定是否开启“设置 - 隧道 - 包括 APNs”，具体来说如果是纯 WiFi 环境则无需开启，但使用 SIM 卡的蜂窝网络时 iOS 可能倾向于更底层的直连路由 APNs 流量，因此需要开启上述设置。设置完成后启用模块，然后开启关闭飞行模式一次（这是网传的方法，根本目的是彻底关闭再开启一次箭箭火隧道），即可复活 APNs。

> 如果你是从[上一篇文章](./10.箭箭火+捷徑：按应用分流.md)的脚注跳过来的，那我再说两句：原本正是利用了 APNs 的无需开启代理就能收到消息的便利性，才能实现不使用手机时关闭箭箭火来省电（也就是箭箭火自带的、我在上篇文章中划掉的“睡眠时断开”），现在需要一直开启了；但如果你不需要复活 APNs 的话倒是不影响。

代理 APNs 会引发一些问题：假设 PROXY 组节点全挂了，那么 APNs 也挂了，国内 App 的消息也收不到；其次 APNs 挂代理本身就会增加延迟。还是建议关闭 iPhone 对应 App 的通知（或卸载它们。既然 Apple 会主动屏蔽这些软件的 APNs，就有理由怀疑它会和 GFW 分享哪些设备在使用这些软件），使用其他设备接收信息吧（物理分流）。硬要用的话，还是参考[上一篇文章](./10.箭箭火+捷徑：按应用分流.md)结合应用分流使用比较合适（这也是为啥我写成模块，因为这样可以无视使用的配置生效，但实测使用体验很差，因为切换配置可能会打断 APNs 的长连接）。
