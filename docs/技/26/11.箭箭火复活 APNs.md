[APNs 是啥，APNs 又咋了？](./10.箭箭火+捷徑：按应用分流.md/#fn:4)

给出一[模块](https://raw.githubusercontent.com/eepsjo/s/refs/heads/m/apns)用于代理 APNs 流量。

模块的原理基于笔者对 APNs 屏蔽的技术原理的*猜测*：笔者*认为*，APNs 通过检测用户的 IP，再根据应用的 ID 来定向屏蔽消息推送。模块通过代理 APNs 的域名和 IP，让 APNs 服务器认为我们在境外，即和 APNs 服务器沟通的 IP 不是东大的。

~~笔者也曾考虑过使用 MITM，在 APNs 直连的前提下欺骗服务器给我们发消息，但这与域前置等仅绕过审查不代理等用法的场景不同：那些场景是目的服务商没有主动屏蔽，而这个场景是 APNs 的主动屏蔽，因此必须代理。~~

以下是使用模块时需要注意的点：

1. 模块使用 `fallback` 代理组代理 APNs（这个代理组的名字是 `APNS`），需要在模块的“编辑参数”中手动填入一个或多个代理节点或分组的名称，每个节点或分组之间用英文逗号分隔。默认值 `PROXY,DIRECT`，建议填写一个或多个稳定、低延迟的节点，然后在末尾填写 `DIRECT`。

    > 手动指定固定的节点，目的是防止切换配置时、或 PROXY 组本身会自己切换时，打断 APNs 的长连接导致信息接收延迟。<br>填写直连作为回落兜底，是防止节点全部超时导致 APNs 完全失效（至少直连时不需要绕过 APNs 屏蔽的应用还能收到消息）。

2. 精简模式是把前文所述的 APNs 的 IP（参考 [Apple 的官方文档](https://support.apple.com/en-us/102266#:~:text=assigned%20to%20Apple.-,If%20you,-can%27t%20allow%20access)，这里说明了 APNs 可能用到的 IP 段）恢复直连，即仅代理 `+.push.apple.com` 域名。尚不明确是否需要代理这些 IP。

3. 根据自己的网络环境决定是否开启“设置 - 隧道 - 包括 APNs”，具体来说，如果是纯 Wi-Fi 环境无需开启，但使用蜂窝网络时需要。

模块配置完成后，开启关闭飞行模式一次（这是网传的方法，根本目的是彻底关闭再开启一次箭箭火隧道）即可复活 APNs。
