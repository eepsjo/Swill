# fake-ip 与 redir-host

## redir-host

``` mermaid
sequenceDiagram
    autonumber
    participant App as App/Browser
    participant Core as 客户端代理内核
    participant Remote as 服务端代理内核

    rect rgb(30, 40, 50)
    Note right of App: 阶段 1: DNS 解析
    App->>Core: 1. 查询 google.com IP
    Core->>Remote: 2. 远程 DNS 解析 (规避污染)
    Remote-->>Core: 3. 返回真实 IP (1.2.3.4)
    Core-->>App: 4. 返回真实 IP (1.2.3.4)
    end

    rect rgb(40, 50, 60)
    Note right of App: 阶段 2: 建立连接
    App->>Core: 5. 连接到 1.2.3.4
    Note over Core: 匹配 Mappings 表: <br/>1.2.3.4 -> google.com
    Core->>Remote: 6. 转发请求 (带上域名 google.com)
    Remote-->>App: 7. 返回数据
    end
```

## fake-ip

fake-ip 的概念早在 2001 年就被提出。随着近年来移动互联网和内核级虚拟网卡（TUN/TAP）的普及，开发者才逐渐意识到“等待 DNS 解析”成为了性能瓶颈和隐私漏洞。

fake-ip 有极致的响应速度。既然代理工具在握手阶段可以只需要域名（让服务端解析以防污染并优化境外流量），那么解析真实 IP 并返回就成了多余的动作。fake-ip 将“解析”与“连接”解耦，在本地建立一套虚拟映射。

``` mermaid
sequenceDiagram
    autonumber
    participant App as App/Browser
    participant Core as 客户端代理内核
    participant Remote as 服务端代理内核
    
    rect rgb(30, 40, 50)
    Note right of App: 阶段 1: 劫持 DNS 响应
    App->>Core: 1. 查询 google.com IP
    Note over Core: 核心逻辑: <br/>不请求 DNS，直接从 fake-ip 池分配<br/>映射 198.18.0.1 <-> google.com
    Core-->>App: 2. 返回 Fake IP (198.18.0.1)
    end

    rect rgb(40, 50, 60)
    Note right of App: 阶段 2: 建立连接与解析
    App->>Core: 3. 向 198.18.0.1 发起连接
    Note over Core: 识别 fake-ip，还原域名 google.com
    Core->>Remote: 4. 转发请求 (告知服务端解析域名)
    Remote-->>App: 5. 返回数据
    end
```

## 对比
|  | fake-ip | redir-host |
| :--- | :--- | :--- |
| DNS | 零延迟，无需等待解析，直接返回映射地址 | 显著延迟。需等待物理 DNS 解析返回结果 |
| 隐私保护 | 极高。本地不产生明文 DNS 请求，完全规避 ISP 监控 | 低。本地存在真实的 DNS 查询流 |
| 分流逻辑 | 先验逻辑。基于域名映射 | 后验逻辑。拿到 IP 后再分流 |

> 现代代理客户端往往有优秀的乐观 DNS 机制，DNS 延迟问题得到改善。

## 其他
在 fake-ip 下，用户常纠结于无法正常 `ping` 域名。但对需要代理的域名而言，ICMP 延迟已失去参考价值，因为握手和数据交换是在远端服务器完成的。

以 mihomo 为例，通过 `fake-ip-filter` 绕过的直连域名，依然可以正常进行物理 `ping` 探测，因为它们完全不进入 fake-ip 池；启用 `disable-icmp-forwarding`，可以把 `ping` 命令重新定义为“检测 fake-ip”的工具，而非传统意义上的 ICMP 探测器。
