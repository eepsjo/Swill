# 浅谈约束工程

> Harness Engineering：LLM 落地生产生活的关键

约束工程（我自己翻的）是一个很大的概念，简单来说目前的 Agent = Model + Harness。“Harness”原意是马具或约束装置，这里指在 LLM 之外所有能决定系统稳定交付的工程化设施。我把它分为 5 层架构：

1. 边界层：明确 LLM 扮演的角色、角色的目标、任务成功执行的标准，并对上下文信息进行裁剪和组织，防止信息过载导致 LLM “幻觉”或遗漏；

    > 熵减的起点。正如“融入集体会失去自我”，边界层的作用正是为 LLM 划定“自我”的绝对范围，防止其在海量信息中发生“人格崩塌”。

2. 工具层：解决给什么工具、何时调用工具以及提炼工具返回结果的问题。

    > 遵循剃刀原则，不仅仅是简单地把 API 扔给 Agent。

3. 执行层：为 LLM 建立任务路径，让它准确理解目标、判断信息、生成、检查、修正的逻辑串联。

4. 记忆层：按需分离当前任务状态、中间结果和长期记忆，防止 Agent 因“失忆”在长任务链路中跑偏；

    > 可以理解成 Neuro-sama 的备份，因为长任务已经接近 Neuro 的长期记忆需求，这里因为失忆产生的问题与“幻觉”类似，但前者主要由初始变量（边界问题）导致，这里主要是 LLM 长任务链路中累积的熵增引发。

5. 评估层：让 Agent 具备独立于对话之外的验证能力，知道自己做得好不好；在失败（如 API 超时、格式错误）时，有自动重试、切换路径或回滚的机制。
   
    > $Text \to Action \to Feedback \to Correction \to Fallback$

注意到约束工程包括了早期 LLM 的两大工程阶段：提示词工程（Prompt Engineering）和上下文工程（Context Engineering），前者是 LLM 刚刚发展起来时为了解决“模型有没有听懂”的问题的工程，通过角色设定、风格约束等塑造局部的概率空间，让模型表达更准确（也就是 AI 心理学）；后者是词元（Token）概念提出后为了解决“信息给得对不对”的问题的工程，核心是检索增强生成（RAG），即在合适的时间把正确的外部知识和状态送入模型。约束工程的目标，是解决“任务能否持续做对”的问题：当 AI 从“回答问题”转向“执行长链路任务”时，需要一套系统来驾驭、约束和纠偏系统。用人话说，约束工程解答了“为什么同样的模型，在不同团队手里表现出的稳定性和成功率天差地别”等类似疑惑。

我在 [Agent 危机](../../文/26/7.Agent 危机.md)一文中指出，如果把 LLM 看作驾驶员，用户的电脑看作飞机，那么形如 OpenClaw 的系统就是驾驶舱；现在从约束工程的角度出发，这类约束系统不仅仅是驾驶舱，还是驾驶指南、操作逻辑、航空时刻图、与塔台（用户）沟通的桥梁等等，而飞机本身也将抽象到具体的任务而非硬件。

> 这么说就是要引入“容器”、“虚拟化”的概念：好的约束工程不应停留在任务和知识的约束上，更应该体现在实际生产环境的权限分配、资源分配和空间分配上。

目前的 Agent 与其说是助理，不如说是研究生，而且你还得给它们发工资（词元、API、硬件、维护、后勤等支出，以及耗费的精力）。这也是为什么约束工程被提出的根本原因：它并不完全是一个工程概念，也不是一套解决方案，而是在 Agent 技术发展的今天针对行业内落地的痛点问题提出的系统设计概念，而在这个概念上具体如何操作才是真正的约束工程。这里给出几个设想：

1. 当上下文太长导致 LLM “焦虑”丢细节时，不只是压缩信息，而是启动一个“干净”的新 LLM 进行任务交接（类似进程重启）；

2. 验收机制独立化。简单来说，将“干活的（Generator）”和“验收的（Evaluator）”分开，让 Evaluator 在“真实”环境（如沙盒、虚拟机）中执行真实操作来验证结果，而非只看代码；

    > 比较有争议的一点是是否应该给 Agent 可变（且可由自己改变）的环境？

3. 让工程师不再写逻辑代码，而是设计环境本身，将巨大的操作方式改为目录索引，按需暴露子文档，防止模型注意力涣散。人话，使 Vibe Coding 中的 Vibe 成为工作核心。

    > Infrastructure as a Vibe (IaaV)。<br>1. 动态沙盒机制：Agent 应该不能也不被允许直接操作物理系统，而是在为其生成的容器镜像中执行。<br>2. 注意力索引：与其塞入大量 Token，不如将环境设计成层级化的空间（Levels ~~后室来的~~），Agent 的“注意力”移动到哪个位置，才激活该位置的文档和工具引用。

AI 发展的重心正从“让模型看起来更聪明”转向“让模型在真实世界里稳定工作”，也就是从聊天的伙伴变成数字劳动力。由于 Agent 往往需要花费更多的 Token 等成本，也解决了我在 [Agent 危机](../../文/26/7.Agent 危机.md)一文中的担忧：AI 技术虽然早已民用，但核心的、转化成生产力的约束工程短期不会大范围普及，避免 AI 沦入“‘未有效参与高阶级生产生活’和‘前沿技术平民化’两大泡沫”中。

但普及是不可避免的，因为普及的本质是使用成本和学习成本的双下降。Agent 会像互联网一样普及吗？答案是肯定的，且速度会比互联网快得多。 互联网解决的是信息连接，而 Agent 解决的是能力杠杆。

> 支点越稳，杠杆越长，人作为施力点的一端，其初始动能（原始知识储备）决定了产出的量级。<br>基础信息的获取将变得廉价，但“提出正确约束逻辑”的能力将成为新的阶级竞争力。如果开发者只掌握 Vibe 而不理解底层的计算机科学逻辑，那么当约束系统本身出现故障时，他们将束手无策。

我们仍需要继续学习掌握传统的计算机知识，小到搜索引擎的使用，大到全栈开发能力，简单来说**不能因为 Agent 的普及而失去主动接受知识的热情和能力**。信息时代最本质的竞争力不是便捷、舒适、自动化，而是信息本身，谁掌握最多的信息、谁拥有最新的技术，谁才是真正的赢家。
