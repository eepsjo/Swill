# 浅谈代理泄露

> 代理泄露是一个很大的概念，但最终是“使用代理”的事实被其他人/组织/平台知晓。本文将从多个角度剖析代理泄露的本质、表现和后果，并给出可能的解决方案。

先说结论：任何分流都无法完美解决代理泄露的问题，除非物理分流（不同设备/虚拟空间配置不同的网络环境，执行不同的网络操作）。

代理，简单来说就是让一台服务器转发你的流量到终点，起到保护客户端的作用。东大语境下大多数人口中所谓的“代理”指的则是利用 GFW 外的服务器等实现绕过 GFW 审查访问国际互联网的行为，即“魔法”、“科学上网”等。因此，代理的网络流程如下：

> $客户端 \to （ISP/GFW \to 流量未被拦截） \to 代理服务端 \to 目标网络$

注意到流程中除了客户端和服务端，还有 ISP、GFW，甚至可能有 CDN 等第三方。现代互联网广泛使用 TLS 加密使得数据可以相对安全的通过它们，但 SNI 明文等问题仍然会导致流量被 ISP/GFW 拦截。本文不讨论代理协议的发展历史，只是就这个流程指出流量可能被监控、篡改、拦截的事实。因此，代理泄露给谁的问题就解释清楚了。

---

## 1. 泄露给 ISP/GFW
这大概是最可怕的泄露，网上广为流传的“DNS 泄露”就是最经典的一种泄露给 ISP/GFW（以下简称 GFW）的代理泄露。

DNS 泄露的原因很简单，由于错误的分流配置或分流中规则过老/更新不及时，导致本来应该代理的流量路由到了直连，使得 DNS 查询记录暴露给 GFW。这种泄露的可怕之处是现代代理客户端的表现行为可能让用户完全不会察觉泄露的发生，即即使 DNS 泄露也不影响代理使用（早期的唱歌盒子甚至利用 GFW 会返回污染 IP，而这些 IP 往往是国外 IP（geoip:!cn）的特性来分流）。

> 注意到上文讨论的是用户在日常使用的机器（有国产 APP/国内网站的流量）上开启全局代理的情况，本文也默认用这个情况讨论。<br>同时，文章仅讨论泄露发生的事实，至于泄露后知晓者如何处理本文不做猜测。

解决 DNS 泄露的方法也很简单，就结论而言笔者倾向 fake-ip。

## 2. 本地泄露
在某些权限隔离不完善的平台，客户端启用的诸如 TUN、fake-ip 服务有明显特征；甚至于某些代理客户端默认开启一个默认端口、无鉴权的 socks 本地代理，设备内的其他进程可以直接探测这些默认的地址来判断设备是否使用了代理。

另一种情况则复杂一点。假设用户配置的分流中最后一条命中所有流量的规则是代理，那么国内 APP 可以通过请求无 SNI 的流量获取这个规则的出口，即代理服务器的 IP。美团已被证实会利用此方法，结合用户在软件中填写的收货地址等把用户和境外 IP 绑定，并建立社工库。

如果仅仅是解决无 SNI 问题可以用嗅探器、IP 屏蔽规则等解决，或者把兜底规则换成直连。但是这样做并不能避免以下两种情况：

1. 黑名单投毒。因为兜底直连意味着前面会配置一些黑名单域名，使他们代理，只要 GFW 在这些社区维护的规则中恶意放入钓鱼域名，就能轻松获取境外 IP。

觉得你的规则没问题？看看第二条，

2. CDN API 泄露。尝试请求`https://chatgpt.com/cdn-cgi/trace`，由于你的黑名单肯定有 ChatGPT，这条流量必须走代理，而该网站托管在 Cloudflare CDN 上，利用 URL 中通用的 API，你的落地 IP 会准确显示出来。国内 APP 可以直接请求这个 URL 来获取你的境外 IP。

因此兜底直连的意义不大。

> 本质上代理的使命是包含客户端安全，因此兜底代理才是正确的行为，这样能确保所有未知流量不会直接攻击客户端，同时也避免 GFW 把某些平常因为被屏蔽无法访问的境外上游服务的 SNI 作为钓鱼蜜罐的可能。

解决这个泄露的方法复杂得多，最好是完全的环境隔离（时空意义上）；退而求其次是基于应用分流（国内应用流量使用仅屏蔽广告的分流，只有需要代理的进程添加代理规则。但是该方案无论是从占用的资源还是现有客户端支持程度的角度来看都不现实）~~；最后是别用国内软件~~。

## 3. 泄露给 CDN
Cloudflare 确实是大善人，但不代表它绝对可信。其他 CDN 也是如此。

简单来说，过 CDN 的流量把 TLS 加密交给了 CDN 处理，即使这些流量使用了 TLS1.3 或 PQC，CDN 总能解密它们，若 CDN 与 GFW 合作，或 CDN 遭受网络攻击导致密钥泄露等，都会导致你的流量被曝光。解决方法是在 TLS 内部加密，或使用安全的传输层协议。

> 嵌套加密可能会有 TiT 特征（Trojan）或增加加解密负担（Vmess）。比较火的 Vless ENC 宣称在其配套自研协议 XTLS/Reality 中可以实现内核级流量转发，同时支持 PQC，但 Reality 本身有更大的问题（鹦鹉已死）且本文仅讨论代理泄露，此处不展开。

有时 CDN 并不欢迎自己成为代理中转器，特别是用于非法访问时，所以代理泄露给 CDN 也可能让它关停你的服务。

## 4. 泄露给目标网络
很多时候打不开某些网站，或卡人机验证或锁区就是这个问题。但是相比于上面的泄露，此泄露的严重性不高，解决方法也简单。

不频繁切换节点、使用高质量的节点；良好的保号措施；客户端使用透明全局代理。

## 5. 其他泄露行为

比如不正确的鉴权处理、在 GFW 外评头论足、在 GFW 内宣扬知识炫耀技术等。

> 比较有争议的是去广告。代理客户端的其他用法就是改写流量或屏蔽流量，如 shadowrocket 的模块可以实现很多本来很复杂的操作；而屏蔽广告在国内 APP 看来确实是异常（好吧国外也是如此但是 who cares?），这可能是很明显的代理特征。但用合法合规的软件如 ADG 也是这样的行为。

---

可以看到随着层级的下降，解决方案的复杂度（物理隔离 - 切换协议 - 调整客户端 - 现实中别说）和问题的严重性也在下降。未来的实践方案可能倾向 CDN + ECH，实现即使软件知道你在使用代理，也无法封禁 IP 和 SNI。但我们当然希望大隐隐于市朝，最终目的并非苟延残喘，而是销声匿迹。不相信任何三方、时刻保持警惕、不分享技术、时刻掌握环境动向，都是保持代理环境不失联的重点。
