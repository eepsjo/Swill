# 咪猫之重新比赛

> [重新比赛](https://wiki.metacubex.one/config/proxies/rematch/)是[咪猫 1.19.28](https://github.com/MetaCubeX/mihomo/releases/tag/v1.19.28) 的新特性（出站）。本文通过两个示例简单分析这个出站的用法。

`rematch` 的核心思想是给流量打标签，并使其回环（Loopback）到 `rules` 模块的开始，也就是进行多轮流量分流，这使得传统的一站式分流逻辑更灵活也更复杂。具体来说 `rematch` 有两个功能，一个是把流量*直接*指向子规则（本质是把标签和子规则联系起来，使其无需 `SUB-RULE` 这个路由规则）；另一个是把流量送回规则匹配伊始，用 `REMATCH-NAME` 这个配套路由规则二次或多次分流。

下文示例用到的规则集：

```yaml
rule-providers:
  R: {type: http, behavior: domain, format: yaml, url: "https://anti-ad.net/clash.yaml", path: ./rule/R.yaml, <<: [*update]}
  r: {type: http, behavior: classical, format: text, url: "https://raw.githubusercontent.com/eepsjo/0/refs/heads/0/r", path: ./rule/r.txt, <<: [*update]}
  d: {type: http, behavior: classical, format: text, url: "https://raw.githubusercontent.com/eepsjo/0/refs/heads/0/d", path: ./rule/d.txt, <<: [*update]}
  p: {type: http, behavior: classical, format: text, url: "https://raw.githubusercontent.com/eepsjo/0/refs/heads/0/p", path: ./rule/p.txt, <<: [*update]}
  T: {type: http, behavior: domain, format: yaml, url: "https://raw.githubusercontent.com/Loyalsoldier/clash-rules/refs/heads/release/tld-not-cn.txt", path: ./rule/d/T.yaml, <<: [*update]}
  G: {type: http, behavior: domain, format: yaml, url: "https://raw.githubusercontent.com/Loyalsoldier/clash-rules/refs/heads/release/gfw.txt", path: ./rule/d/G.yaml, <<: [*update]}
  D: {type: http, behavior: domain, format: yaml, url: "https://raw.githubusercontent.com/Loyalsoldier/clash-rules/refs/heads/release/direct.txt", path: ./rule/p/D.yaml, <<: [*update]}
  I: {type: http, behavior: ipcidr, format: yaml, url: "https://raw.githubusercontent.com/Loyalsoldier/clash-rules/refs/heads/release/cncidr.txt", path: ./rule/p/I.yaml, <<: [*update]}
```

## target-sub-rule

兜底代理时为实现 PROXYblockQUIC 不能用 `SUB-RULE`，因为 `MATCH` 无法使用子规则。

```yaml
rules:
  - RULE-SET,R,REJECT
  - RULE-SET,r,REJECT
  - RULE-SET,d,DIRECT
  # 这里可以
  - SUB-RULE,(RULE-SET,p),PROXYblockQUIC
  - RULE-SET,D,DIRECT
  - RULE-SET,I,DIRECT,no-resolve
  # 这里不能用，只能写成两行
  - AND,((NETWORK,UDP),(DST-PORT,443)),REJECT
  - MATCH,PROXY

sub-rules:
  PROXYblockQUIC: ['AND,((NETWORK,UDP),(DST-PORT,443)),REJECT', 'MATCH,PROXY']
```

有了 `rematch` 就可以

```yaml
rules:
  - RULE-SET,R,REJECT
  - RULE-SET,r,REJECT
  - RULE-SET,d,DIRECT
  # 下一条规则的 PbQ 匹配的是下方 proxies 模块的内联 rematch 节点（出站）。MATCH 同
  - RULE-SET,p,PbQ
  - RULE-SET,D,DIRECT
  - RULE-SET,I,DIRECT,no-resolve
  - MATCH,PbQ

sub-rules:
  PROXYblockQUIC: ['AND,((NETWORK,UDP),(DST-PORT,443)),REJECT', 'MATCH,PROXY']

proxies:
  # 这里的意思是内联节点 PbQ，如果在规则中匹配到，则跳转到子规则 PROXYblockQUIC
  - {name: PbQ, type: rematch, target-sub-rule: "PROXYblockQUIC"}
```

## target-rematch-name

示例[^1]利用 `rematch` 实现一键切换兜底模式。

[^1]: 示例中⚠️注释后的子规则有问题：咪猫无法在非 IP 路由规则实现 no-resolve，即使规则集是 IP-CIDR，也只能在路由规则是 RULE-SET 时使用。这样配置不会报错，但也不会生效。如果上面的逻辑规则 AND 直接用 no-resolve 则会报错“不存在这个出站”。

    无法正确使用 no-resolve 会造成 DNS 泄露，因此示例仅作 `rematch` 的功能示范，不建议使用。

> 实际上我写完才发现完全不用这么复杂，真想一键切换直接把所有规则都写好然后决定 `MATCH` 的分流就行，但这么做不能很好决定不同兜底的规则触发顺序。`rematch` 则可以做到特定兜底只匹配特定规则。

```yaml
proxy-groups:
  - {name: 节点}
  #
  - {name: MATCH, type: select, proxies: [D, P]}
  #
  - {name: GLOBAL, type: select, proxies: [PROXY], hidden: true}
  - {name: PROXY, type: select, proxies: [节点]}

rules:
  # 下面四个是不管什么兜底都要加载的规则
  # 事实上 d 和 p 谁先触发也有讲究，真正可以在 REMATCH-NAME 之前触发的应该只有 R 和 r。这里简化处理
  - RULE-SET,R,REJECT
  - RULE-SET,r,REJECT
  - RULE-SET,d,DIRECT
  - RULE-SET,p,PbQ
  # 第一轮规则匹配时，这里会直接跳转到最下面的 MATCH
  # 如果 MATCH 组选择了 D（兜底直连）。第二轮可以看作从这里开始匹配，下同
  - AND,((REMATCH-NAME,d),(RULE-SET,T)),PbQ
  - AND,((REMATCH-NAME,d),(RULE-SET,G)),PbQ
  # 第二轮匹配的终点，相当于 MATCH。下同
  - REMATCH-NAME,d,DIRECT
  # 如果 MATCH 组选择了 P（兜底代理）
  - AND,((REMATCH-NAME,p),(RULE-SET,D)),DIRECT
  # ⚠️
  - AND,((REMATCH-NAME,p),(RULE-SET,I)),DnR
  - REMATCH-NAME,p,PbQ
  # 第一轮匹配的终点，到这里的流量都会因为 MATCH 组流入二轮
  - MATCH,MATCH

sub-rules:
  PROXYblockQUIC: ['AND,((NETWORK,UDP),(DST-PORT,443)),REJECT', 'MATCH,PROXY']
  # ⚠️
  DIRECTnoResolve: ['MATCH,DIRECT,no-resolve']

proxies:
  - {name: D, type: rematch, target-rematch-name: "d"}
  - {name: P, type: rematch, target-rematch-name: "p"}
  - {name: PbQ, type: rematch, target-sub-rule: "PROXYblockQUIC"}
  # ⚠️
  - {name: DnR, type: rematch, target-sub-rule: "DIRECTnoResolve"}
